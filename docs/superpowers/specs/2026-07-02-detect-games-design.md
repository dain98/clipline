# Detect Games Design

## Goal

Add a manual Detect Games workflow to Settings > Games so users can scan the machine for likely games, choose unchecked candidates from a list, and add the selected rows into the existing Custom games list.

## Decision

Clipline will not auto-add or auto-record newly discovered games in this feature. Discovery is an explicit user action, every detected row starts unchecked, and selected rows become normal `CustomGameSettings` entries only after the user clicks Add game(s).

The first implementation should combine two discovery sources:

- Steam installed-game manifests, to find games that are installed even when they are closed.
- Currently visible capturable windows, to find running games and to upgrade Steam candidates with exact process path, executable name, window title, and icon when possible.

This keeps the feature useful without claiming that Steam manifests alone reveal the final executable launched by every game.

## User Experience

Settings > Games keeps the existing Supported games and Custom games sections. In the Custom games header, add a Detect Games button to the left of Add Custom Game.

Pressing Detect Games opens an inline panel under the Custom games list. The panel shows a loading state while the backend scans, then displays a dense list of detected candidates. Each row has:

- an unchecked checkbox.
- the candidate icon when available, otherwise the same placeholder treatment as Custom games.
- the game name.
- a source label such as Steam, Running window, or Steam + running window.
- concise metadata, preferring `exe_name - window_title` when a running window is known, otherwise the install path or Steam app id.

The panel footer has Add game(s) and Cancel actions. Add game(s) is disabled until at least one candidate is checked. Adding selected rows appends them to the existing Custom games draft, hides or refreshes the detection panel, and updates the settings status with save-to-apply copy. The user still saves settings through the existing Save button.

## Candidate Model

Add a backend-facing detected-game candidate shape that is intentionally close to `CustomGameSettings` but not persisted directly:

```rust
pub struct DetectedGameCandidate {
    pub id_hint: String,
    pub name: String,
    pub source: DetectedGameSource,
    pub steam_app_id: Option<u32>,
    pub install_dir: Option<String>,
    pub exe_name: String,
    pub process_path: Option<String>,
    pub window_title: String,
    pub icon: Option<String>,
    pub confidence: u16,
}
```

The frontend converts checked candidates into `CustomGameSettings` with:

- `id`: generated by the existing custom-game id helper, using the candidate name.
- `name`: candidate name.
- `enabled`: `true`.
- `exe_name`: candidate executable name when known.
- `process_path`: candidate process path when known.
- `window_title`: candidate window title when known.
- `recording_mode`: `replays_only`.
- `icon`: candidate icon when available.

Backend validation must still require a match identity. Steam-only candidates appear in the checkbox list only when discovery has inferred a usable executable identity. Steam apps without a safe executable inference are skipped for this version.

## Discovery Architecture

Add a focused discovery module under the app crate, for example `apps/clipline-app/src/game_discovery.rs`.

Responsibilities:

- Locate Steam libraries on Windows using registry and common install fallbacks.
- Parse `libraryfolders.vdf` and `appmanifest_*.acf`.
- Produce Steam installed-game records with app id, display name, install directory, and source metadata.
- Infer Steam-only executable identity with a bounded install-directory scan that filters helper tools and prefers executable names close to the Steam app name or install folder.
- Read currently visible capturable windows through the existing game/window enumeration boundary.
- Merge Steam records with running windows when the window process path sits under the Steam install directory.
- Add running-window candidates that are not matched to Steam so non-Steam games remain discoverable.
- Remove launcher/helper noise with a small denylist for browsers, Steam/Epic clients, crash handlers, uninstallers, redistributables, and obvious update helpers.

The module should stay anti-cheat-safe: normal filesystem reads, registry reads, visible window metadata, process executable path metadata, and icon extraction only. No injection, no memory reads, no game-process hooks, and no privileged scanning.

## Tauri Commands

Add a command named `detect_installed_games` that returns detected candidates. Mark it async so the scan does not block the UI thread. The command should be best-effort:

- Steam unreadable or missing returns candidates from running windows.
- Individual malformed manifests are skipped.
- Icon extraction failure leaves `icon` as `None`.
- Unexpected scan errors are surfaced as a concise user-facing error only when the full scan cannot continue.

Keep the existing `list_game_windows` and Add Custom Game picker unchanged.

## Duplicate Handling

The detection result should avoid duplicates before rendering:

- Existing custom games are considered duplicates by normalized process path, then executable name, then lowercased name.
- Built-in supported games can still appear only if they would be added as a custom rule for the actual executable. If this creates confusing duplicates with supported profiles, prefer hiding those candidates.
- Steam and running-window records for the same game merge into one row.
- Multiple windows from the same process collapse to the best row, preferring rows with process path, icon, and longer non-launcher title.

The frontend should also guard against duplicates when adding selected rows, because settings may have changed since the scan was produced.

## Data Flow

1. User clicks Detect Games.
2. `settings.js` opens the detection panel and invokes `detect_installed_games`.
3. Rust scans Steam manifests and visible windows, merges candidates, removes obvious duplicates/noise, and returns sorted candidates.
4. The UI renders unchecked candidate rows.
5. User checks rows and clicks Add game(s).
6. The frontend converts checked candidates into normal custom game draft entries.
7. `renderCustomGames()` refreshes the existing Custom games list.
8. User clicks Save, and the existing settings validation and recorder restart path apply the new custom games.

## Sorting

Sort results for scan readability:

1. Running candidates with exact process path.
2. Steam candidates upgraded by a running window.
3. Steam candidates with executable identity inferred safely.
4. Alphabetical by display name inside each group.

## Testing

Backend tests should cover:

- Parsing Steam `libraryfolders.vdf` fixtures.
- Parsing Steam `appmanifest_*.acf` fixtures.
- Handling malformed manifests without failing the whole scan.
- Merging a Steam app with a running window whose process path is under the install directory.
- Keeping a running non-Steam window as its own candidate.
- Filtering obvious launcher/helper/browser candidates.
- Deduping against existing custom game identities.

Frontend/UI contract tests should cover:

- `index.html` contains `detect-games`, `detected-games-panel`, `detected-games-list`, and Add game(s)/Cancel controls.
- `settings.js` invokes `detect_installed_games`.
- Detect Games opens the panel, renders unchecked candidates, disables Add game(s) until selection, and appends selected rows through the existing Custom games draft flow.
- Existing custom game rendering and Add Custom Game picker wiring remain intact.

Manual verification should cover:

- Clean workspace build and `cargo test --workspace`.
- Opening the app and pressing Detect Games with Steam installed.
- Adding one running Steam game and confirming it appears in Custom games with icon, executable metadata, Replays only, and enabled checked.
- Adding a running non-Steam window through Detect Games.
- Confirming duplicate detections do not create duplicate custom games.

## Non-Goals

This feature does not add automatic recording for newly discovered games, background library scanning, Epic/GOG/Xbox providers, cloud game catalogs, Steam appinfo binary parsing, per-game recording-mode selection inside the detection panel, or any game-specific event marker integrations.
